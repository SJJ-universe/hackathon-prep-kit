해커톤 부트킷 (Windows 원샷 개발환경)

새 노트북이나 대회 제공 PC(둘 다 Windows)에서 코딩 도구를 한 번에 설치하고,

에이전트(Claude Code / Codex)가 이 저장소를 열자마자 동일한 방식으로 바이브코딩을 시작하게 만드는 키트입니다.

에이전트 최초 실행 (필수)

이 저장소를 열면 다른 무엇보다 먼저 아래 한 줄을 실행해 모든 도구를 한 번에 설치하라.

설치가 끝나기 전에는 코드 작업을 시작하지 말 것.

로컬(이미 클론했거나 파일을 받은 경우):

powershell -NoProfile -ExecutionPolicy Bypass -File ./setup.ps1

원격(아직 클론 안 함 / 빈 PC — 이 저장소 기준 URL 그대로 실행):

irm https://raw.githubusercontent.com/SJJ-universe/hackathon-prep-kit/main/setup.ps1 | iex

비개발자/더블클릭용: bootstrap.cmd 를 더블클릭해도 동일하게 설치됩니다.

설치 후 검증 섹션에서 git / node / npm / python / gh / netlify 등의 버전이 출력되면 정상입니다.

일부 CLI가 NOT FOUND 면 새 터미널을 열고(PATH 갱신) 다시 확인하세요.

처음 켠 PC라 에이전트(Claude CLI)조차 없다면: 먼저 0로 Claude CLI를 설치·인증·실행한 뒤, 그 에이전트에게 위 한 줄(setup.ps1)을 맡기세요. 사람이 직접 setup.ps1을 돌려도 Claude를 포함한 전체가 설치됩니다.

04. 에이전트 부트스트랩 — 새 PC에서 Claude Code(우선)·Codex 0부터 첫 실행까지

새 노트북이나 대회 제공 PC(Windows)에서 코딩 에이전트를 0부터 설치하고, 첫 실행에 성공시키는 단계별 가이드입니다. 제일 먼저 Claude CLI를 띄우는 흐름을 1순위로 두고, 그다음 그 에이전트에게 setup.ps1을 맡겨 나머지 도구를 한 번에 까는 순서로 설계했습니다.

이 문서의 사실(패키지명·URL·최소 버전·인증 절차)은 2026-06 기준 웹 검색으로 확인했습니다. 확실하지 않은 항목은 본문에 미확인으로 표기했습니다.

0단계 — 선행 확인 (2분)

새 PowerShell 창을 열고 아래를 차례로 확인합니다.

확인 항목명령기대 결과 / 조치
인터넷 연결Test-NetConnection api.anthropic.com -Port 443TcpTestSucceeded : True. 실패하면 사내 프록시/방화벽 문제(트러블슈팅 표 참조)
관리자 여부([Security.Principal.WindowsPrincipal][Security.Principal.WindowsIdentity]::GetCurrent()).IsInRole('Administrator')True면 시스템 설치 가능. False여도 user 스코프로 대부분 진행 가능

winget이 없을 때: Microsoft Store에서 App Installer를 설치하면 winget이 생깁니다. Store 페이지 https://apps.microsoft.com/detail/9NBLGGH4NNS1. 설치 후 새 터미널을 엽니다.

Node 설치 (Claude Code 필수 전제)

Claude Code는 npm 설치 경로를 쓰려면 Node.js 18 이상이 필요합니다(공식 npm 패키지 배지·설치 가이드 기준 "Node.js 18+"). 없거나 구버전이면 winget 한 줄로 LTS를 깝니다.

winget install -e --id OpenJS.NodeJS.LTS --accept-package-agreements --accept-source-agreements

설치 후에는 새 터미널을 열어야 node / npm이 PATH에 반영됩니다. 확인:

node --version
npm --version
참고: 네이티브 설치 스크립트(아래 1단계 (c))를 쓰면 Node 없이도 Claude CLI가 동작합니다. 다만 이 키트의 setup.ps1은 npm 경로로 Claude를 깔므로 Node가 있어야 setup이 매끄럽습니다.

1단계 — Claude CLI 설치 (가장 먼저)

세 가지 경로가 있습니다. 상황에 맞게 하나만 고르면 됩니다. 공식 npm 패키지명은 @anthropic-ai/claude-code(확인됨)입니다.

(a) 권장 — setup.ps1 원샷에 포함되어 자동 설치

이 키트의 setup.ps1이 Node·Git 등과 함께 Claude CLI를 npm으로 깔고 끝에 버전까지 검증합니다. 손이 가장 적게 갑니다.

powershell -NoProfile -ExecutionPolicy Bypass -File ./setup.ps1
  • 장점: 도구 전체를 한 번에. 멱등(여러 번 실행해도 안전). 버전 검증 요약 출력.
  • 단점: Node 등 다른 설치까지 같이 도므로 시간이 더 걸림. 인증은 자동으로 안 됨(3단계 수동).
  • 닭-달걀 주의: setup.ps1을 "에이전트에게 시키려면" 에이전트(Claude)가 이미 떠 있어야 합니다. 그래서 첫 한 번은 사람이 아래 (b) 또는 (c)로 Claude만 먼저 깔고 → Claude를 띄운 뒤 → 그 Claude에게 setup.ps1을 맡기는 흐름을 권장합니다.

(b) 단독 설치 — npm (사람이 Claude만 먼저 깔 때 1순위)

Node가 이미 있으면 이 한 줄이 가장 빠릅니다.

npm install -g @anthropic-ai/claude-code
  • 장점: 빠르고 단순. 버전 고정(@latest, 특정 버전 핀)에 유리. 사내 npm 미러와 호환.
  • 단점: Node 18+ 필요. 전역 bin(%APPDATA%\npm)이 PATH에 없으면 claude를 못 찾음(2단계 참조). 업데이트는 수동(npm i -g @anthropic-ai/claude-code@latest).

(c) 대안 — 네이티브 설치 스크립트 (Node 불필요)

Node를 깔기 싫거나 막힐 때. Windows PowerShell용 공식 원라인(확인됨):

irm https://claude.ai/install.ps1 | iex

(참고 — macOS/Linux는 curl -fsSL https://claude.ai/install.sh | bash)

  • 장점: Node 의존성 없음. 백그라운드 자동 업데이트. 의존성 제로 설치.
  • 단점: 설치 위치를 자체 user-local bin에 깔고 PATH에 추가하므로 새 터미널이 필요. 그 bin의 정확한 경로는 환경마다 다를 수 있음(미확인 — 보통 사용자 프로필 하위). 사내 프록시 환경에서 스크립트 다운로드가 막히면 (b) npm 경로로 우회.

2단계 — PATH / 실행정책

PATH: claude를 못 찾을 때

npm 전역 설치(-g)는 실행 파일을 %APPDATA%\npm(보통 C:\Users\<사용자>\AppData\Roaming\npm)에 둡니다. 이 폴더가 PATH에 없으면 설치는 됐어도 claude 명령을 못 찾습니다.

  1. 가장 흔한 원인은 "설치한 그 창"이 옛 PATH라서 그렇습니다. 새 터미널을 열면 대개 해결됩니다.
  2. 현재 PATH에 들어있는지 확인:
   $env:Path -split ';' | Select-String npm
  1. 없으면 user PATH에 영구 추가(새 터미널부터 적용):
   setx PATH "$($env:Path);$env:APPDATA\npm"

(이미 들어있으면 중복 추가하지 말 것. setx는 현재 창에 즉시 반영되지 않으니 새 창에서 확인.)

  1. npm이 보는 전역 위치 직접 확인:
   npm config get prefix
   npm root -g

네이티브 설치((c))로 깐 경우엔 설치기가 자체 bin을 PATH에 추가하므로, 마찬가지로 새 터미널을 연 뒤 claude --version을 확인합니다. 그래도 안 보이면 npm 경로((b))로 재설치하는 게 가장 확실합니다.

PowerShell 실행정책 (ExecutionPolicy)

setup.ps1 같은 .ps1 스크립트는 기본 정책에서 차단될 수 있습니다. 두 가지 방법:

  • 일회성(권장, 시스템 설정 안 바꿈): 스크립트를 실행할 때만 우회.
  powershell -NoProfile -ExecutionPolicy Bypass -File ./setup.ps1
  • 영구(현재 사용자만, 안전한 절충): 서명된 원격 스크립트는 막고 로컬은 허용.
  Set-ExecutionPolicy -Scope CurrentUser RemoteSigned

차이: Bypass는 아무 것도 막지 않음(해당 프로세스에만 한정해 쓰면 안전). RemoteSigned는 인터넷에서 받은 미서명 스크립트는 막고 로컬에서 작성한 건 허용 — 일상 개발용 영구 설정으로 적당합니다. 회사 정책으로 Set-ExecutionPolicy가 막혀 있으면, 일회성 -ExecutionPolicy Bypass 방식만 쓰면 됩니다.

3단계 — 인증

claude를 깔았다고 바로 동작하지 않습니다. 한 번 인증해야 합니다. 두 방법 중 하나를 고릅니다.

방법 A — 대화형 로그인 (구독 사용 시 권장)

claude를 처음 실행하면 브라우저가 열리고 Anthropic 계정으로 로그인하게 됩니다.

claude
  • Claude Pro/Max 등 구독으로 쓰려면 이 방법을 쓰고, ANTHROPIC_API_KEY는 설정하지 마세요. 환경변수에 API 키가 있으면 Claude Code는 구독이 아니라 API 종량제로 과금합니다(공식 지원 문서 기준). 현재 인증 상태는 실행 중 /status로 확인합니다.
  • 대회장 공용 PC라면, 끝나고 로그아웃(또는 claude 자격 정리)을 잊지 마세요.

방법 B — 환경변수 ANTHROPIC_API_KEY (API 계정/무인 실행 시)

구독 없이 API 키로 쓰거나, 로그인 창을 띄울 수 없는 자동화 환경이면 환경변수로 키를 박습니다. setx로 넣으면 다음에 여는 터미널부터 영구 적용됩니다.

setx ANTHROPIC_API_KEY "sk-ant-여기에-발급받은-키"
  • 키 발급처: Anthropic Console — https://console.anthropic.com (API Keys 메뉴). 결제수단이 등록된 Console 계정이 필요합니다(무료 claude.ai 플랜으로는 Claude Code 사용 불가).
  • 주의: setx는 현재 창에는 반영되지 않습니다. 넣은 뒤 새 터미널을 열고 echo $env:ANTHROPIC_API_KEY로 확인하세요.
  • 주의: 위 방법 A(구독)와 충돌합니다. 구독으로 쓸 거면 이 변수를 비워 두세요. 제거는 setx ANTHROPIC_API_KEY "" 후 새 터미널(또는 시스템 환경변수 편집에서 삭제).

node --version
npm --version

  • 함정: 한글 PDF가 단일 리스크 → 30분 게이트로 티어0 성공 또는 티어1 확정. "법률사무 아님"(문서 미작성·죄명 무단정) 30초 선언. "안전" 라벨 출력 불가(거짓 안심 금지).

3) 골든타임 (통과③, 노벨티 가장 깨끗)

  • 1장면: "어제 잘렸고 애 둘에 월세" → 사건·발생일·가구 추출 + 제도가 마감 임박순으로 한 화면 정렬.
  • 체크리스트: LLM 키 · 15090532/15108347/15113968(자동승인) 캐시 200~300건 + 라이브 1건 · 법령 마감 룩업표 · Web Speech.
  • 함정: 마감 숫자는 AI 생성 금지·법령 큐레이션만(조문 스냅샷). 상대시간("어제") 해소가 AI load-bearing 증명. 선정기준 산문 여부는 당일 캡처로 확정(아니면 사건 칩 폴백).

4) 히트네임 폭염 (앱, 라이브 사수 안정)

  • 1장면: 내 위치 체감온도 → 폭염 N단계 + 대상별 행동카드 + 지금 운영중 쉼터.
  • 체크리스트: 기상청 15084084(자동승인) · 격자 변환 코드 · 무더위쉼터 15013199(위경도 당일확인) · (선택)Geolocation.
  • 함정: 격자(nx,ny) 변환 누락이 1순위 버그. 쉼터 좌표 오류·야간 미운영 → 운영시간 필터. 단순 쉼터지도는 네이버맵과 중복 → 개인 맞춤 위험지수로 차별화.

5) 웨이크520 화재 (앱, 무설치 최강)

  • 1장면: 경보음 트리거 → 화면 점멸 + 진동 + 520Hz 구형파 + "불이야 대피하세요" 음성 + 큰 '화재' 글자.
  • 체크리스트: Web Audio(오실레이터) · Vibration · speechSynthesis. 외부 데이터·키·라이브러리 0.
  • 함정: 마이크 상시청취의 배터리·오탐·iOS 진동 미지원을 데모에서 정직하게(데모는 톤 트리거로). 하드웨어 보급은 한국이 두꺼움 → "폰만으로 깨우기 증폭"으로 포지셔닝.

6) 디코드 재난문자 (앱)

  • 1장면: 재난문자 붙여넣기 → 쉬운 한국어·다국어 요약 + 거주형태별(반지하 등) 정반대 행동 + 가까운 대피소.

7) 리콜체크 / 바코드 리콜 (앱+고득점, 단일 최강 후보)

6-B. 요점

  • 대부분의 MVP는 추가 설치 0(공통 스택 + CDN 한 줄 + 키 발급)으로 만들어진다. practice-apps 10종은 외부 라이브러리·지도 SDK·라이브 API 없이 바닐라 + 내장 샘플 + 서비스워커만으로 작동했다.
  • 무거운 별도 설치가 필요한 경우만 별도 표기 대상이다: MediaPipe(모델·카메라, 낙상 코치) · Python/FastAPI·Pillow(로컬 런타임). 그 외(jsPDF·html2canvas·qrcode·MediaPipe까지)는 전부 CDN으로 끝난다.
  • 키는 사전 발급이 안전선: data.go.kr(자동승인 즉시)는 당일도 가능하나, safetydata·식품안전나라·Kakao·VWorld·Claude는 승인대기·결제등록이 있어 대회 전 발급해 .env에 둔다(저장소 커밋 금지).
  • 라이브 사수 1순위 = 기상청 단기예보 15084084(자동승인·표준 REST). 지오코딩 1순위 = Kakao Local. 둘로 "실제로 도는 것"을 증명하고 나머지는 캐시 샘플+더미로 운영한다.
  • CORS·키노출이 걸리면 FastAPI/Node 미니 프록시 1파일을 당일 작성(완성 앱 코드 아님)한다. 정적·캐시 샘플로만 시연하면 프록시 없이도 성립한다.
  • 좌표 함정 2개(기상청 격자 nx,ny / 대피소 도분초 DMS)는 변환 코드를 미리 숙지한다.

03 호스팅·심사위원 시연 — 3경로 (빠른 순서)

심사 시점에 "실제 작동·시연 가능"이 인정 조건이다(목업·발표자료만은 불인정). 심사위원에게 보여주는 방법을 빠른 순서로 3개 준비한다. 1순위 Netlify Drop, 막히면 GitHub Pages, 그래도 안 되면 로컬 폴백. 시연 직전 점검과 QR은 맨 아래.

Netlify의 현재 화면 기준 단계별 상세(드롭 1시간 클레임·CLI v26·고정 슬러그·SPA 리다이렉트·Functions로 키 숨기기·무료 크레딧 한도·카카오맵 도메인 등록)는 06 Netlify 배포 상세 가이드를 본다. 폴백 다음 칸(Vercel 미러 L1·cloudflared 터널 L2)은 07 미러·터널 배포 상세 가이드.

전제: 보여줄 산출물은 정적 파일 묶음(빌드형이면 npm run build의 dist, 정적 HTML이면 그 폴더)이다. 서버사이드 API 프록시가 있으면 그 부분은 호스팅이 별도로 필요하다(아래 각 경로의 "프록시 주의" 참고).

경로 1 — Netlify Drop (무인증·가장 빠름) [1순위]

브라우저에서 https://app.netlify.com/drop 을 열고, 배포할 폴더를 통째로 드래그&드롭한다(zip은 먼저 압축 해제, index.html은 폴더 루트). 수 초 안에 https://<랜덤이름>.netlify.app URL이 발급된다. 발표용은 먼저 로그인 후 드롭하라 — 무로그인으로 만든 사이트는 1시간 내 클레임하지 않으면 소멸한다(공식: "claim within one hour").

절차:

  1. 산출물 폴더를 확정한다. 빌드형: npm run build → dist. 정적: index.html이 들어있는 폴더.
  2. 그 폴더를 app.netlify.com/drop에 드래그. (zip으로 올려도 됨)
  3. 발급된 https URL을 복사 → 다른 기기에서 즉시 확인.
  4. (선택) 파일을 고쳤으면 같은 페이지에 다시 드래그하면 새 URL/갱신. CLI를 쓰면 netlify deploy --dir dist --prod.

예상 함정:

  • 빌드 없는 정적이라 즉시 뜬다는 게 장점. 단 상대경로로 자산을 참조해야 한다(/assets/... 절대경로는 루트 배포라 보통 OK지만, 하위경로 가정 코드는 깨질 수 있음).
  • 캐시: 재배포 후 옛 화면이 보이면 강력새로고침(Ctrl+F5) 또는 시크릿창.
  • HTTPS 필요 기능은 Netlify가 https라 정상 동작(위치정보·카메라·클립보드·서비스워커). 로컬 http와 달리 여기선 막히지 않는다.
  • 프록시 주의: 브라우저에서 외부 공개 API를 직접 부르면 CORS·키 노출 문제가 난다. 키가 필요한 호출은 Netlify Functions(서버리스)로 옮기거나, 시연용으로 사전 캐시한 응답 JSON을 정적으로 함께 올린다(화면엔 출처칩 유지). 별도 FastAPI 프록시를 쓰는 구조면 그 프록시는 로컬 또는 Vercel/Render 등에 따로 띄우고 프론트의 API base만 그 주소로.

경로 2 — GitHub Pages (gh로 repo + pages) [2순위]

정적 사이트를 GitHub에 올리고 Pages로 공개한다. URL은 https://<아이디>.github.io/<저장소>/ 형태. 반영에 1~2분 걸린다.

평이한 정적(루트에 index.html) 절차:

gh auth status                                  # 로그인 확인 (사전 완료)
gh repo create <repo> --public --source=. --push
# Pages 켜기 (main 브랜치 루트)
gh api --method POST repos/<아이디>/<repo>/pages -f "source[branch]=main" -f "source[path]=/"
# 또는 GitHub 웹 Settings > Pages 에서 Source=main /(root) 선택

빌드형(vite 등) 절차:

# vite.config 에 base 설정 필요 (하위경로 배포라서)
#   export default { base: '/<repo>/' }
npm run build
npx gh-pages -d dist        # dist를 gh-pages 브랜치로 푸시 (gh-pages 패키지)
# Pages Source = gh-pages /(root)

예상 함정:

  • base 경로: 프로젝트 페이지는 https://id.github.io/repo/ 하위라, 자산을 절대경로(/assets)로 박으면 404. vite는 base: '/repo/', 순수 HTML은 상대경로(./assets/...) 사용. (사용자 페이지 id.github.io는 base가 /라 이 문제 없음)
  • 반영 지연: 켠 직후 404는 정상, 1~2분 기다린 뒤 새로고침. 첫 공개는 더 걸릴 수 있음.
  • SPA 라우팅 404: 클라이언트 라우터를 쓰면 새로고침 시 404 — 해시 라우팅을 쓰거나 404.html을 index.html 복사본으로 둔다. 단일 화면이면 무관.
  • 프록시 주의: GitHub Pages는 정적 전용(서버 없음). 키 필요한 호출은 외부 프록시로 빼거나 사전 캐시 JSON 사용(경로 1과 동일).
  • public repo: 무료 Pages는 public 권장. 키·.env가 커밋되지 않았는지 확인(.gitignore).

경로 3 — 로컬 폴백 (같은 와이파이 또는 화면공유) [최후]

호스팅이 전부 막혀도 시연은 가능하다. 발표 PC에서 로컬 서버를 띄우고, 같은 와이파이의 심사 기기에서 접속하거나 화면공유로 보여준다.

서버 띄우기 — 부트킷의 serve.ps1(있으면) 또는 직접:

.\serve.ps1                      # 있으면 (포트·디렉터리 기본값으로 기동)
# 없으면 둘 중 하나:
npx http-server ./dist -p 8080   # 정적/빌드 산출물
python -m http.server 8080       # 파이썬만 있을 때

같은 와이파이에서 접속:

ipconfig                         # IPv4 주소 확인 (예: 192.168.0.12)
# 심사 기기 브라우저에서:  http://192.168.0.12:8080

예상 함정:

  • 방화벽: 첫 실행 시 Windows 방화벽 허용 프롬프트가 뜨면 "예"(사설 네트워크 허용). 막히면 접속 안 됨.
  • HTTPS 필요 기능: 위치정보·카메라 등은 보안 컨텍스트가 필요. localhost는 예외로 http에서도 동작하지만, http://<IP>로 다른 기기에서 열면 막힌다. → 그 기능을 보여줄 땐 발표 PC 자신의 http://localhost:8080 + 화면공유/미러링으로 시연한다.
  • 대회장 와이파이 격리: 게스트망은 기기 간 통신을 막는 경우가 많다. 이때는 같은 와이파이 접속을 포기하고 화면공유(빔/HDMI/미러링)로 전환.
  • 백업: 정상 플로우 30초 녹화·스크린샷 3장을 PC에 두고, 라이브가 흔들리면 즉시 재생.

시연 직전 체크 (무대 오르기 전 60초)

  • [ ] 다른 기기(휴대폰/다른 노트북)에서 URL을 직접 열어 봤다
  • [ ] 그 기기에서 새로고침(강력) 했고 옛 화면 캐시가 아니다
  • [ ] 핵심 기능 1장면(심사위원 즉석입력 → 실호출 → 출처칩)이 그 자리에서 돈다
  • [ ] HTTPS 필요 기능을 쓴다면 그 기능이 시연 경로에서 막히지 않는다(경로별 함정 확인)
  • [ ] degraded 토글·사람연결 폴백이 뜬다 / 큰글씨·음성·언어전환이 켜진다
  • [ ] URL과 QR을 슬라이드/종이에 띄워 심사위원이 직접 자기 폰으로 열어볼 수 있다
  • [ ] 백업(30초 녹화·스크린샷) 재생 준비됨

발표 동선·역할 분담은 0, 통과 필수 급소는 1.

QR 코드 만드는 법 (한 줄)

  • 오프라인(추천·망 불안 대비): npx qrcode "https://<배포URL>" -o qr.png → 슬라이드/종이에 삽입.
  • 온라인(빠를 때): qr-code-generator류 사이트에 URL 붙여넣기 → PNG 저장. (현장 망이 불안하면 오프라인 생성이 안전)

06 Netlify 배포 상세 가이드 (현재 사이트 기준, 2026-06 검증)

심사위원이 자기 폰으로 QR을 찍어 직접 실행하려면 결과물을 공개 HTTPS 주소에 올려야 한다. Netlify는 그 일을 가장 빠르게(드래그 한 번) 해 주는 정적 호스트다. 이 문서는 현재(2026-06) Netlify 화면·명령 기준의 단계별 사용법이다. 빠른 요약은 03 호스팅·심사위원 시연에 있고, 여기서는 그 1순위(L0) 경로(Netlify)를 끝까지 펼친다. 다음 폴백 칸인 Vercel 미러(L1)·cloudflared 터널(L2)은 07 미러·터널 배포 상세 가이드를 본다.

핵심 사실(검증): 무료 플랜은 2025-09-04부터 크레딧제로 바뀌었고 초과 결제가 없는 하드캡이다(소진 시 사이트 일시정지). 데모 당일 멈춤을 피하려면 큰 미디어·무거운 Functions를 피하라. UI 메뉴가 "Sites"에서 "Projects"로 개명되었다(옛 튜토리얼의 "Site settings" = 지금의 "Project configuration").

0. 두 경로 — 언제 무엇을 쓰나

경로 A: Netlify Drop경로 B: Netlify CLI
방식브라우저에 폴더 드래그터미널 netlify deploy --prod
사전 준비없음(무인증 가능) / 로그인 권장netlify-cli 설치 + 로그인(또는 토큰)
속도수 초첫 로그인 포함 1~2분
무대 적합현장 1순위(가장 단순·실패지점 적음)반복 재배포·자동화·에이전트에 유리
영속성로그인 안 하면 1시간 후 소멸로그인 계정에 영구 보존

권장: 현장 본배포는 Drop(로그인한 상태로), 연습·자동 재배포는 CLI. 둘 다 준비해 두고, 무대에서는 Drop을 1순위로 둔다.

경로 A — Netlify Drop (현재 화면 단계별)

A-1. 먼저 로그인부터 (중요)

무인증으로도 드롭은 되지만, 만들어진 사이트는 1시간 안에 클레임(회원가입/로그인으로 귀속)하지 않으면 사라진다(공식 문구: "claim within one hour"). 발표 전에 만든 데모 URL이 발표 도중 사라지면 치명적이다. 그러니:

  1. https://app.netlify.com 에서 먼저 로그인(GitHub 계정으로 가입이 가장 빠름).
  2. 그 다음에 드롭한다 → 사이트가 내 팀에 영구 귀속된다.

A-2. 드롭 절차

  1. https://app.netlify.com/drop 를 연다(또는 Projects 대시보드 하단의 드롭존).
  2. 폴더를 통째로 드래그한다. (zip 아님 — zip이면 먼저 압축을 풀고 폴더를 드래그. 낱개 파일도 아님.)
  • 빌드형이면 빌드 산출 폴더(dist / build / public)를 드래그.
  • 무빌드 정적이면 index.html이 든 폴더.
  1. index.html이 폴더 최상위(루트)에 있어야 한다. 하위폴더 안에 있으면 "Please drop a folder containing an index.html file" 오류 또는 배포 후 404가 난다(가장 흔한 실수).
  2. 드롭하면 수 초 내 라이브. https://<랜덤단어>.netlify.app 형태의 URL이 발급된다.

브라우저/환경: 최신 Chromium 계열 브라우저, 안정적 네트워크, 그리고 압축 해제 기준 프로젝트 용량의 약 4배 여유 RAM을 권고한다.

A-3. 용량 한도 (드래그앤드롭)

  • 전체 배포: 50MB 미만일 때 가장 잘 된다.
  • 개별 파일: 10MB 초과 파일은 배포가 멈출 수 있다.
  • 이를 넘으면 Drop 대신 CLI(경로 B)를 쓰라고 Netlify가 안내한다. (한글 폰트 base64·동영상 등 큰 자산이 있으면 주의 — 폰트는 서브셋, 동영상은 외부 링크/유튜브 권장.)

A-4. 고정된 URL로 이름 바꾸기 (현재 메뉴 경로)

랜덤 URL은 QR·카카오맵 도메인 등록에 불편하다. 공유 전에 한 번 이름을 바꿔 고정 슬러그를 만든다.

  • 현재 경로: 사이트 클릭 → Project configuration → General → Project details → "Manage project name" (구 라벨: Site configuration → Site details → "Change site name").
  • 입력 후 저장하면 즉시 https://<새이름>.netlify.app 로 바뀐다.
  • 주의: 옛 랜덤 URL은 새 주소로 리다이렉트되지 않는다. 그러니 QR을 만들고 공유하기 전에 먼저 개명할 것.

A-5. 업데이트 재배포 (같은 사이트 유지)

코드를 고친 뒤에는 그 사이트의 Deploys 페이지 하단 드롭존에 새 산출 폴더를 다시 드래그한다(Production deploys 영역). 같은 URL/슬러그를 유지한 채 새 배포가 시작된다.

  • 함정: 일반 app.netlify.com/drop 페이지에 다시 드롭하면 새 사이트(새 랜덤 URL)가 생긴다 — 업데이트할 때는 그러지 말 것.

경로 B — Netlify CLI (현재 명령 기준)

B-1. 설치

npm install -g netlify-cli
  • 현재 버전대: netlify-cli v26.x, Node.js 20.12.2 이상 필요(키트 setup.ps1의 Node LTS면 충족).
  • 실행 바이너리 이름은 netlify. 설치 확인: netlify --version.
  • 전역설치가 막히면 일회성으로 npx netlify-cli <명령> 도 가능.

B-2. 인증 — 두 방식

대화형(브라우저 OAuth):

netlify login

브라우저가 열려 "Netlify CLI" 권한을 허용하면 토큰이 로컬에 저장된다(Windows: AppData\Roaming\netlify\Config\config.json).

비대화형(CI / 에이전트 — 브라우저 못 띄울 때): 환경변수로 토큰을 주면 netlify login을 건너뛴다.

# PowerShell
$env:NETLIFY_AUTH_TOKEN = "토큰값"

개인 액세스 토큰(PAT) 발급 경로(현재 UI): User settings → Applications → Personal access tokens (바로가기: https://app.netlify.com/user/applications#personal-access-tokens) → New access token → 이름·만료 설정 → Generate token → 복사해 NETLIFY_AUTH_TOKEN으로 보관.

B-3. 배포 명령 (현재)

netlify deploy                         # 드래프트 배포 -> 임시 미리보기 URL
netlify deploy --prod                  # 프로덕션(실 URL)로 배포
netlify deploy --prod --dir=dist       # 배포할 폴더 지정
netlify deploy --prod --dir=. --no-build   # 무빌드 순수 정적 한 줄
netlify deploy --site <이름또는ID> --prod  # 특정 사이트로 배포

주요 플래그(모두 현재 유효): --prod(프로덕션), --dir(배포 폴더), --site(대상 사이트), --no-build(빌드 단계 생략 — 순수 HTML 폴더에 필수), --alias(예측 가능한 미리보기 URL, 최대 37자), --message, --allow-anonymous(무로그인 배포·드래프트).

첫 실행 때 폴더가 사이트에 연결돼 있지 않으면, 기존 사이트 선택 / 새 사이트 생성을 물어보고 이후 그 폴더를 연결해 둔다.

B-4. 새 사이트에 비대화형으로 올리기 (에이전트/CI)

가장 단순한 길은 링크 단계를 건너뛰고 deploy가 사이트를 만들게 하는 것:

$env:NETLIFY_AUTH_TOKEN = "토큰값"
netlify deploy --prod --dir=. --no-build --site my-demo-name

명시적으로 만들고 올리려면:

netlify sites:create --name my-demo-name
netlify deploy --prod --dir=. --no-build
키트에는 이 경로를 감싼 도우미 scaffold/deploy-netlify.ps1 이 있다. 한 줄로 Drop 페이지를 열거나 CLI 배포를 시도한다(설치·인증 점검 포함). 사용법은 아래 "한 줄 배포" 참고.

SPA 404 방지 (새로고침 시 빈 화면/404)

클라이언트 라우터를 쓰면 하위경로 새로고침 때 404가 난다. 모든 경로를 앱 셸로 보내는 캐치올 리라이트를 둔다(단일 화면 정적이면 불필요).

방법 A — _redirects 파일(확장자 없음). 배포(publish) 폴더 루트에 둔다. Vite/CRA면 public/에 두면 빌드가 산출 폴더로 복사한다.

/*    /index.html   200

방법 B — netlify.toml(저장소 루트):

[[redirects]]
  from = "/*"
  to = "/index.html"
  status = 200

200이면 리라이트(주소창 URL 유지, SPA 라우터가 처리). 이 파일이 실제 배포된 산출물 안에 들어가야 효과가 있다(저장소에만 있고 빌드에 안 들어가면 무효). Drop으로 올릴 때도 드래그하는 폴더 안에 _redirects가 있어야 한다.

고정/커스텀 도메인 + 카카오맵 도메인 등록 경고

  • 커스텀 도메인: Project overview → Domain management → Add a domain → "Add a domain you already own"(또는 구매).
  • 고정 *.netlify.app 서브도메인: A-4의 이름 변경으로 확정(공유 전 한 번).
  • 카카오맵 직격 주의: Kakao JavaScript SDK는 웹 도메인 화이트리스트를 강제한다. 최종 배포 오리진(예: https://my-demo-name.netlify.app 또는 커스텀 도메인)을 Kakao Developers → 내 애플리케이션 → 앱 설정 → 플랫폼 → Web → 사이트 도메인에 등록해야 한다. 랜덤 미리보기 서브도메인과 개명/커스텀 도메인은 서로 다른 오리진이다 — 실제 배포할 도메인을 등록하지 않으면 심사위원 폰에서 지도가 조용히 "도메인 미등록" 오류로 깨진다. localhost(개발용)와 배포 도메인을 둘 다 등록해 두는 것이 안전.

Netlify Functions — API 키 숨기기 / CORS 프록시

브라우저에서 외부 공개 API를 직접 부르면 CORS 또는 키 노출 문제가 난다. 키가 필요한 호출은 서버리스 함수 뒤로 숨긴다(정적 사이트 그대로 유지하면서).

  • 폴더: netlify/functions/ (기본).
  • 프런트 호출 경로: /.netlify/functions/<함수이름>.
  • 현재 시그니처(중요 변경): 옛 export const handler = async (event) => { return { statusCode, body } }(AWS Lambda 스타일)는 폐지 예정(2027-07-01 이후 배포 거부). 2026 기준 권장형:
// netlify/functions/proxy.mjs
export default async (req, context) => {
  const key = process.env.MY_API_KEY;          // 서버에서만 읽힘(브라우저 노출 안 됨)
  const upstream = await fetch("https://api.example.com/data?key=" + key);
  const data = await upstream.json();
  return new Response(JSON.stringify(data), {
    headers: {
      "content-type": "application/json",
      "access-control-allow-origin": "*",
    },
  });
};

프런트: fetch('/.netlify/functions/proxy') — 키가 브라우저에 닿지 않고, 함수 응답이 CORS도 해결한다.

  • 환경변수 설정 경로(현재): Project configuration → Environment variables(구 "Site configuration"). 함수에서 process.env.MY_API_KEY로 읽는다. 절대 프런트 번들에 키를 넣지 말 것.
  • 무료 플랜 함수는 동기 실행 10초 제한. 무거운 작업/대용량 응답은 피한다.
더 단순한 대안: 키가 필요 없는 시연이라면, 데모용 응답을 사전 캐시 JSON(/mock/<입력>.json)으로 번들에 동봉하고 라이브 → 실패 시 캐시로 폴백한다(함수조차 불필요). 라이브 API가 막혀도 심사위원에겐 에러 대신 데이터가 보인다.

07 미러·터널 배포 상세 가이드 — Vercel + cloudflared (현재 기준, 2026-06 검증)

06 Netlify 배포가 1순위(L0)라면, 이 문서는 그 폴백 사다리의 다음 두 칸이다. 라이브 데모의 제1원칙은 "네트워크는 반드시 한 번 고장난다"이고, 그래서 같은 빌드를 다른 벤더에 한 번 더(L1 Vercel 미러) 올려 두고, 둘 다 막히면 내 노트북을 직접 공개(L2 cloudflared 터널)한다. 무대에서는 만들어 둔 것으로 전환만 한다(즉석 제작 금지).

L0  Netlify(공개 정적·HTTPS)        <- 기본 (06 가이드)
L1  Vercel(다른 벤더 같은 빌드)      <- 한 호스트/CDN 장애·방화벽 차단 대비   (이 문서 1부)
L2  cloudflared 터널 + 내 폰 핫스팟  <- 공개 호스트 둘 다 불통/라이브 백엔드   (이 문서 2부)
L3  녹화 MP4(로컬·망 0)             <- 전부 불통 (03 가이드)

빠른 3경로 개요는 03 호스팅·시연을 본다.

1부 — Vercel 미러 (L1)

1-1. 왜 "미러"인가

L0(Netlify)와 같은 산출물을 서로 무관한 두 번째 벤더(Vercel)에 한 번 더 배포해 둔다. DNS·CDN·장애 범위가 달라, 한쪽 벤더 장애나 대회장 방화벽이 *.netlify.app를 막아도 다른 URL은 살아 있다. 무대에는 A/B 두 QR 카드를 들고, A가 안 뜨면 B를 건넨다(수 초 전환).

1-2. 설치

npm install -g vercel
vercel --version
  • 현재 버전대 vercel 54.x, Node 18 이상(키트 setup.ps1의 Node LTS면 충족).
  • 명령 이름은 vercel(별칭 vc). 인자 없이 vercel = vercel deploy.

1-3. 인증 — 두 방식

대화형:

vercel login            # 브라우저/이메일
vercel login --github   # 깃허브로

비대화형(CI/에이전트): 토큰을 환경변수로.

# PowerShell
$env:VERCEL_TOKEN = "토큰값"
vercel --prod --yes --token $env:VERCEL_TOKEN

토큰 발급 경로(현재): Account Settings → Tokens (https://vercel.com/account/tokens). CLI로도 vercel tokens add "이름".

1-4. 순수 정적 폴더 배포 (빌드 없음)

# 폴더 안에서
vercel                  # 프리뷰(미리보기) 배포 -> 임시 URL
vercel --prod           # 프로덕션 배포 -> 고정 *.vercel.app

# 완전 비대화형(프롬프트 기본값 수락)
vercel --prod --yes

# 폴더로 안 들어가고
vercel --cwd .\dist --prod --yes
  • --prod 없으면 프리뷰 URL(라이브처럼 보여도 고정 도메인 아님). 미러용은 반드시 --prod.
  • 프레임워크 없는 순수 index.html은 Vercel이 그대로 정적 서빙한다(별도 vercel.json 불필요).
  • 첫 배포는 (--yes 안 쓰면) 팀 선택 → "기존 프로젝트 연결?"(N) → 프로젝트 이름 → 루트 → 빌드설정 오버라이드(N)를 물은 뒤, 폴더에 .vercel/ 링크 폴더를 만든다. 이후 배포는 이를 재사용.

1-5. 고정/커스텀 URL

  • 기본 URL: https://<프로젝트이름>.vercel.app — 프로젝트 이름이 곧 서브도메인(선착순, 예약 불가).
  • 이름 변경(서브도메인 변경): 대시보드 → 프로젝트 → Settings → General → Project Name.
  • 커스텀 도메인: Settings → Domains → apex(A) 또는 서브도메인(CNAME). CLI: vercel alias set <배포URL> <도메인>.

1-6. 드래그앤드롭 "Vercel Drop"

  • https://vercel.com/drop (2026-06-12 신설). 파일/폴더/zip을 드롭 → 팀 선택 → 프로젝트 이름 입력 → Deploy(바로 프로덕션).
  • 로그인 필요(팀·이름 선택 때문). 정적은 그대로, 프레임워크는 자동 빌드.
  • 함정: 드롭할 때마다 새 프로젝트가 생긴다(기존 프로젝트로 재배포 아님). 반복 갱신은 CLI를 쓰라.

1-7. SPA 라우팅 (새로고침 404 방지)

vercel.json(저장소 루트):

{
  "$schema": "https://openapi.vercel.sh/vercel.json",
  "rewrites": [{ "source": "/(.*)", "destination": "/index.html" }]
}

단일 화면 정적이면 불필요. (키트 스캐폴드에 vercel.json 템플릿 포함.)

1-8. 무료 "Hobby" 한도 + 비상업 조항 (주의)

  • 월 포함량(현재): Fast Data Transfer ≈ 100GB/월, 배포 100회/일, 함수 호출 100만, 프로젝트 200개 등. 초과는 과금 대신 기능이 약 30일 일시정지(Hobby).
  • 비상업 조항(원문): "the Hobby plan restricts users to non-commercial, personal use only." 즉 Hobby는 비상업·개인용이다.
  • 해커톤 회색지대: 수익 없는 데모/포트폴리오는 전형적 Hobby 용도지만, 스폰서·고객·수익화와 엮이면 "상업"으로 간주되어 원칙적으로 Pro($20/월)가 필요하다. 미러 용도로 잠깐 쓰는 비수익 데모면 통상 문제 없으나, 상업성이 있으면 미러는 Netlify에 맡기고 Vercel은 피하거나 Pro로.

1-9. 함정 (2025–2026)

  • .vercel/ 링크 폴더는 .gitignore에 넣는다(첫 배포 시 생성). 다른 프로젝트로 재연결하려면 삭제하거나 vercel link. 스테일 링크는 엉뚱한 프로젝트로 조용히 배포한다.
  • --name 플래그는 폐지 → 프로젝트 링크/--yes 기본값 사용.
  • 프리뷰 vs 프로덕션: 맨 vercel은 프리뷰. 미러는 vercel --prod.
  • 파일이 수천 개면 vercel deploy --archive=tgz.

2부 — cloudflared 터널 (L2)

2-1. 왜 "터널"인가

공개 호스트(L0·L1)가 둘 다 막히거나, 내 노트북에서 도는 라이브 백엔드(프록시/LLM 서버)를 그대로 보여야 할 때, 내 노트북의 로컬 서버를 공개 HTTPS 주소로 즉시 노출한다. 핵심: 대회장 와이파이가 아니라 내 폰 핫스팟에 노트북을 물려 자기 회선으로 서빙한다(대회장 와이파이는 데모 최대 사망원인).

왜 cloudflared인가(= ngrok이 아닌가): ngrok 무료는 브라우저 접속 시 강제 "Visit Site" 경고 페이지(인터스티셜)를 띄우고 가입/계정을 요구한다. cloudflared Quick Tunnel은 무가입·무로그인·인터스티셜 없음이라 심사위원 폰에서 바로 열린다.

2-2. 설치 (Windows)

winget(권장, ID 확인됨 Cloudflare.cloudflared):

winget install -e --id Cloudflare.cloudflared

직접 다운로드(패키지매니저 없이):

# 단독 실행 .exe (폴더에 두고 그 자리에서 실행, 또는 PATH 추가)
curl.exe -L -o cloudflared.exe https://github.com/cloudflare/cloudflared/releases/latest/download/cloudflared-windows-amd64.exe
# 또는 MSI 설치본(PATH 자동 등록): .../cloudflared-windows-amd64.msi
  • choco/scoop도 있음: choco install cloudflared / scoop install cloudflared.
  • 주의: cloudflared는 Windows에서 자동 업데이트 안 됨(필요 시 수동).

2-3. Quick Tunnel 명령

cloudflared tunnel --url http://localhost:8137
  • 계정·로그인·DNS 불필요. 랜덤 https://<랜덤>.trycloudflare.com URL을 터미널에 출력(보통 10초 이내).
  • 함정: .cloudflared 폴더에 config.yaml이 있으면 Quick Tunnel이 시작되지 않는다 → 데모 때는 설정 파일 없는 상태로 실행.

2-4. 전체 흐름 (Windows, 터미널 2개 — 둘 다 열어 둬야 함)

# 터미널 1 — 빌드된 정적 폴더를 고정 포트로 서빙
npx serve ./dist -l 8137
#   (Node 없이:)  python -m http.server 8137 --directory dist
# 터미널 2 — 그 포트를 HTTPS로 공개
cloudflared tunnel --url http://localhost:8137

출력된 https://<랜덤>.trycloudflare.com을 심사위원에게 전달(QR). 터널이 HTTPS(실제 Cloudflare 인증서)로 종단하므로 보안 컨텍스트가 성립 → 카메라(getUserMedia)·위치(geolocation)가 심사위원 폰에서 작동한다. (이게 http://192.168.x.x LAN 공유 대신 터널을 쓰는 결정적 이유 — LAN http에선 이 API들이 조용히 죽는다.)

2-5. 데모 내내 살아있게 유지

  • 두 터미널 창을 끝까지 열어 둔다. 하나라도 닫히면 서버(터미널1) 또는 공개 URL(터미널2)이 죽는다.
  • 절전/화면꺼짐 차단(관리자 PowerShell, AC 전원):
powercfg /change standby-timeout-ac 0
powercfg /change monitor-timeout-ac 0
  • 덮개 닫기 동작을 "아무 것도 안 함"으로(제어판 > 전원 옵션 > 덮개를 닫으면). AC 전원 유지. (절전/종료 시 URL이 새로 바뀐다.)

2-6. 캐비엇 (Cloudflare 문서 기준)

  • 테스트/개발용: "Quick Tunnels are intended for testing and development only." 프로덕션 SLA 보장 없음.
  • 휘발성 URL: 프로세스가 살아있는 동안만 유효. 재실행하면 새 랜덤 URL.
  • 동시성 한도 200 in-flight: 초과 시 429. 심사위원 몇 명엔 충분하나 부하테스트 금지.
  • SSE 미지원: Server-Sent Events(EventSource 기반 실시간) 안 됨. (WebSocket은 일반적으로 지원.)

2-7. 핫스팟·QR 메커닉

  • 노트북을 내 폰 핫스팟에 연결(대회장 와이파이 회피). 터널 URL은 망과 무관하게 공개 HTTPS다.
  • QR은 오프라인 생성: npx qrcode "https://<랜덤>.trycloudflare.com" -o qr.png. 단 터널 URL은 매 실행마다 바뀌므로 현장에서 띄운 뒤 그 자리에서 QR 재생성(미리 인쇄 불가).

Static App Scaffold

대회 당일, 빈 새 저장소에서 곧바로 시작할 수 있는 무프레임워크(바닐라) 정적 앱 골격입니다.

주제에 종속되지 않은 일반 템플릿이며, MVP 1~2개 기능만 빠르게 얹는 데 최적화되어 있습니다.

반입 규칙 주의 (중요)

  • 이 스캐폴드는 일반 골격일 뿐, 주제 맞춤 로직은 들어있지 않습니다.
  • 아이디어 구현 코드는 반드시 대회 당일, 빈 새 저장소에서 작성하세요.
  • 파일 안의 TODO 표시 지점에만 로직을 채우는 방식으로 사용합니다.

구성

  • index.html : 헤더 / 메인(입력 영역, 결과 영역) / 푸터. 시맨틱 마크업 + 접근성 기본.
  • app.css : 디자인 토큰(색·간격·타이포·라운드·그림자) + 범용 컴포넌트(버튼·카드·인풋·뱃지·토스트).
  • app.js : 셀렉터 헬퍼, 안전 출력(escape), 간단 상태 store, 결과 렌더 골격, 서비스워커 가드.
  • serve.ps1 : 로컬 정적 서버 실행 + 브라우저 자동 오픈.
  • deploy-netlify.ps1 : Netlify Drop 안내 + CLI 배포(-Dir/-SiteName/-DropOnly/-Draft). 상세는 5.
  • netlify.toml : SPA 404 방지 리다이렉트 + 기본 보안 헤더(제너릭). 빌드형이면 publish만 조정.
  • deploy-vercel.ps1 : Vercel 미러 배포(L1, -Dir/-Prod). 상세는 3.
  • tunnel-cloudflared.ps1 : 로컬 서버 + cloudflared quick tunnel(L2, -Dir/-Port). 공개 HTTPS로 폰에서 카메라/위치 동작.
  • vercel.json : Vercel용 SPA rewrites 템플릿(제너릭).
  • deploy-ghpages.ps1 : gh CLI로 저장소 생성 → 푸시 → Pages 활성화.

빠른 시작 (당일 흐름)

  1. 새 저장소에 이 파일들을 복사한다.
  2. 로컬 확인:
   .\serve.ps1

기본 http://localhost:8137 로 열립니다. (http로 열어야 fetch/서비스워커 정상 동작)

  1. 아이디어 구현은 다음 세 곳의 TODO에만 추가:
  • index.html : 입력 컨트롤, 결과 마크업
  • app.js : submit 핸들러의 [입력 → 계산], renderResult() [결과 표시]
  • app.css : 필요 시 --color-accent 등 토큰만 조정
  1. 배포(둘 중 하나):
   .\deploy-netlify.ps1          # 가장 빠름(브라우저 Drop) + CLI
   .\deploy-ghpages.ps1 -Repo my-demo

MVP 원칙

  • 기능은 1~2개로 좁힌다. "입력 → 계산 → 결과"가 한 화면에서 끝나게 한다.
  • 외부 의존성을 최소화한다(프레임워크/빌드 없음 → 어떤 PC에서도 즉시 실행).
  • 사용자/외부 입력은 항상 escapeHtml로 감싸 안전하게 출력한다.

1줄 데모 원칙

  • 심사위원에게 보여줄 핵심 동작을 한 문장으로 말할 수 있어야 한다.

예: "이 값을 넣으면 이 결과가 즉시 나옵니다." (그 한 동작이 끊김 없이 되게 만든다.)
